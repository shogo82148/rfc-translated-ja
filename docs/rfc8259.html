RFC 8259

The JavaScript Object Notation (JSON) Data Interchange Format

RFC 8259

JavaScript Object Notation (JSON) データ交換フォーマット

Abstract

概要

JavaScript Object Notation (JSON) is a lightweight, text-based, language-independent data interchange format. It was derived from the ECMAScript Programming Language Standard. JSON defines a small set of formatting rules for the portable representation of structured data.

JavaScript Object Notation (JSON) は、軽量でテキストベースの、 言語に依存しないデータ交換フォーマットです。これは ECMAScript プログラミング言語標準に由来しています。JSON は、構造化された データを移植可能に表現するための小さな書式規則の集合を定義します。

This document removes inconsistencies with other specifications of JSON, repairs specification errors, and offers experience-based interoperability guidance.

この文書は、JSON の他の仕様との不整合を取り除き、 仕様上の誤りを修正し、経験に基づく 相互運用性のための指針を示します。

1. Introduction

1. はじめに

JavaScript Object Notation (JSON) is a text format for the serialization of structured data. It is derived from the object literals of JavaScript, as defined in the ECMAScript Programming Language Standard, Third Edition [ECMA-262].

JavaScript Object Notation (JSON) は、構造化データを 直列化するためのテキスト形式です。これは、ECMAScript Programming Language Standard, Third Edition [ECMA-262] で定義されている JavaScript のオブジェクトリテラルに由来します。

JSON can represent four primitive types (strings, numbers, booleans, and null) and two structured types (objects and arrays).

JSON は、4 つの基本型（文字列、数値、真偽値、 および null）と 2 つの構造型（オブジェクトと配列）を表現できます。

A string is a sequence of zero or more Unicode characters [UNICODE]. Note that this citation references the latest version of Unicode rather than a specific release. It is not expected that future changes in the Unicode specification will impact the syntax of JSON.

文字列は、0 個以上の Unicode 文字の並びです [UNICODE]。 この参照は Unicode の特定の版ではなく最新バージョンを参照していることに 注意してください。将来の Unicode 仕様の変更が JSON の構文に影響を与えることは 想定されていません。

An object is an unordered collection of zero or more name/value pairs, where a name is a string and a value is a string, number, boolean, null, object, or array.

オブジェクトは、0 個以上の名前/値の組からなる順序なしの集合であり、 名前は文字列で、値は文字列、数値、真偽値、null、オブジェクト、 または配列です。

An array is an ordered sequence of zero or more values.

配列は、0 個以上の値からなる順序付きの並びです。

The terms "object" and "array" come from the conventions of JavaScript.

"object" および "array" という用語は、 JavaScript の慣習に由来します。

JSON's design goals were for it to be minimal, portable, textual, and a subset of JavaScript.

JSON の設計目標は、最小限で、移植可能で、テキストベースであり、 かつ JavaScript のサブセットであることでした。

1.1. この文書で使用する規則

キーワード「しなければなりません（MUST）」、「してはなりません（MUST NOT）」、 「要求されています（REQUIRED）」、 「することになります（SHALL）」、「することはありません（SHALL NOT）」、 「すべきです（SHOULD）」、「すべきではありません（SHOULD NOT）」、 「推奨されます（RECOMMENDED）」、「推奨されません（NOT RECOMMENDED）」、 「してもよいです（MAY）」、「選択できます（OPTIONAL）」は、 BCP 14 [RFC2119] [RFC8174] に記載されているとおりに解釈されるものとします。 ただし、ここに示すようにすべて大文字で表示される場合に限ります。

1.2. Specifications of JSON

1.2. JSON の仕様

This document replaces [RFC7159]. [RFC7159] obsoleted [RFC4627], which originally described JSON and registered the media type "application/json".

この文書は [RFC7159] を置き換えます。[RFC7159] は [RFC4627] を廃止しました。RFC4627 はもともと JSON を記述し、メディアタイプ "application/json" を登録したものです。

JSON is also described in [ECMA-404].

JSON は [ECMA-404] にも記述されています。

The reference to ECMA-404 in the previous sentence is normative, not with the usual meaning that implementors need to consult it in order to understand this document, but to emphasize that there are no inconsistencies in the definition of the term "JSON text" in any of its specifications. Note, however, that ECMA-404 allows several practices that this specification recommends avoiding in the interests of maximal interoperability.

前文における ECMA-404 への参照は規範的なものです。これは実装者がこの文書を理解するために参照する必要があるという通常の意味ではなく、いずれの仕様においても「JSON テキスト」という用語の定義に矛盾がないことを強調するためです。ただし、ECMA-404 は、最大限の相互運用性の観点からこの仕様が避けることを推奨するいくつかの慣行を許可していることに注意してください。

The intent is that the grammar is the same between the two documents, although different descriptions are used. If there is a difference found between them, ECMA and the IETF will work together to update both documents.

両文書の間では異なる記述が使われていますが、文法は同一であることが意図されています。もし差異が見つかった場合、ECMA と IETF は協力して両文書を更新します。

If an error is found with either document, the other should be examined to see if it has a similar error; if it does, it should be fixed, if possible.

いずれかの文書にエラーが見つかった場合、もう一方の文書にも同様のエラーがないか確認する必要があります。もしあれば、可能であれば修正されるべきです。

If either document is changed in the future, ECMA and the IETF will work together to ensure that the two documents stay aligned through the change.

将来いずれかの文書が変更される場合、ECMA と IETF は協力して変更を通じて両文書の整合性を保つようにします。

1.3. Introduction to This Revision

1.3. この改訂版の紹介

In the years since the publication of RFC 4627, JSON has found very wide use. This experience has revealed certain patterns that, while allowed by its specifications, have caused interoperability problems.

RFC 4627 の公開以来の年月の中で、JSON は非常に広く使用されるようになりました。この経験から、仕様では許可されているものの、相互運用性の問題を引き起こす特定のパターンが明らかになりました。

Also, a small number of errata have been reported regarding RFC 4627 (see RFC Errata IDs 607 [Err607] and 3607 [Err3607]) and regarding RFC 7159 (see RFC Errata IDs 3915 [Err3915], 4264 [Err4264], 4336 [Err4336], and 4388 [Err4388]).

また、RFC 4627 に関して少数のエラータが報告されています （RFC Errata IDs 607 [Err607] および 3607 [Err3607] を参照）。また、RFC 7159 に関しても同様です（RFC Errata IDs 3915 [Err3915]、4264 [Err4264]、4336 [Err4336]、および 4388 [Err4388] を参照）。

This document's goal is to apply the errata, remove inconsistencies with other specifications of JSON, and highlight practices that can lead to interoperability problems.

この文書の目的は、エラータを適用し、JSON の他の仕様との不整合を取り除き、相互運用性の問題につながる可能性のある慣行を明示することです。

2. JSON Grammar

2. JSON 文法

A JSON text is a sequence of tokens. The set of tokens includes six structural characters, strings, numbers, and three literal names.

JSON テキストはトークンの列です。トークンの集合には、6 つの構造文字、文字列、数値、および 3 つのリテラル名が含まれます。

A JSON text is a serialized value. Note that certain previous specifications of JSON constrained a JSON text to be an object or an array. Implementations that generate only objects or arrays where a JSON text is called for will be interoperable in the sense that all implementations will accept these as conforming JSON texts.

JSON テキストはシリアライズされた値です。以前の JSON の仕様では、JSON テキストをオブジェクトまたは配列に限定していたことに注意してください。JSON テキストが求められる場所でオブジェクトまたは配列のみを生成する実装は、すべての実装がこれらを適合した JSON テキストとして受け入れるという意味で相互運用性を持ちます。

   JSON-text = ws value ws


   JSON-text = ws value ws

These are the six structural characters:

以下は 6 つの構造文字です：

   begin-array     = ws %x5B ws  ; [ left square bracket


   begin-array     = ws %x5B ws  ; [ left square bracket


   begin-object    = ws %x7B ws  ; { left curly bracket


   begin-object    = ws %x7B ws  ; { left curly bracket


   end-array       = ws %x5D ws  ; ] right square bracket


   end-array       = ws %x5D ws  ; ] right square bracket


   end-object      = ws %x7D ws  ; } right curly bracket


   end-object      = ws %x7D ws  ; } right curly bracket


   name-separator  = ws %x3A ws  ; : colon


   name-separator  = ws %x3A ws  ; : colon


   value-separator = ws %x2C ws  ; , comma


   value-separator = ws %x2C ws  ; , comma

Insignificant whitespace is allowed before or after any of the six structural characters.

6 つの構造文字のいずれかの前後には、意味を持たない空白文字を使用できます。

   ws = *(
           %x20 /              ; Space
           %x09 /              ; Horizontal tab
           %x0A /              ; Line feed or New line
           %x0D )              ; Carriage return


   ws = *(
           %x20 /              ; Space
           %x09 /              ; Horizontal tab
           %x0A /              ; Line feed or New line
           %x0D )              ; Carriage return

3. Values

3. 値

A JSON value MUST be an object, array, number, or string, or one of the following three literal names:

JSON の値は、オブジェクト、配列、数値、文字列、または以下の 3 つのリテラル名のいずれかでなければなりません（MUST）：

   false
   null
   true


   false
   null
   true

The literal names MUST be lowercase. No other literal names are allowed.

リテラル名は小文字でなければなりません（MUST）。他のリテラル名は使用できません。

   value = false / null / true / object / array / number / string


   value = false / null / true / object / array / number / string


   false = %x66.61.6c.73.65   ; false


   false = %x66.61.6c.73.65   ; false


   null  = %x6e.75.6c.6c      ; null


   null  = %x6e.75.6c.6c      ; null


   true  = %x74.72.75.65      ; true


   true  = %x74.72.75.65      ; true

4. Objects

4. オブジェクト

An object structure is represented as a pair of curly brackets surrounding zero or more name/value pairs (or members). A name is a string. A single colon comes after each name, separating the name from the value. A single comma separates a value from a following name. The names within an object SHOULD be unique.

オブジェクト構造は、掲别符䣱のペアで表現され、ゼロ個以上の名前と値のペア（またはメンバー）を囲みます。名前は文字列です。各名前の後に単一のコロンが続き、名前を値から分香します。単一のカンマは値を後続の名前から分香します。オブジェクト内の名前は一意であるべきです（SHOULD）。

   object = begin-object [ member *( value-separator member ) ]
            end-object


   object = begin-object [ member *( value-separator member ) ]
            end-object


   member = string name-separator value


   member = string name-separator value

An object whose names are all unique is interoperable in the sense that all software implementations receiving that object will agree on the name-value mappings. When the names within an object are not unique, the behavior of software that receives such an object is unpredictable. Many implementations report the last name/value pair only. Other implementations report an error or fail to parse the object, and some implementations report all of the name/value pairs, including duplicates.

名前がすべて一意であるオブジェクトは、相互運用性を持ちます。すべてのソフトウェア実装がこのオブジェクトを受け給りた時、名前と値のマッピングに関して冶一致します。オブジェクト内の名前が一意でない場合、そのようなオブジェクトを受け給るソフトウェアの動作は予測不可能です。多くの実装は最後の名前と値のペアだけをレポートします。他の実装はエラーをレポートしたり、オブジェクトの解析に失敗したりします。また、重複を含るすべての名前と値のペアをレポートする実装もあります。

JSON parsing libraries have been observed to differ as to whether or not they make the ordering of object members visible to calling software. Implementations whose behavior does not depend on member ordering will be interoperable in the sense that they will not be affected by these differences.

JSON 解析ライブラリは、オブジェクトメンバーの顺序を呼び出し元ソフトウェアに詳して表示させるか的について相違いがあることが確認されています。メンバーの顺序に依存しない動作をする実装は、相互運用性を持ちます。これらの違いに影響を受けないという意味です。

5. Arrays

5. 配列

An array structure is represented as square brackets surrounding zero or more values (or elements). Elements are separated by commas.

配列構造は比門括弧で表現され、ゼロ個以上の値（または要素）を囲みます。要素はカンマで分香されます。

array = begin-array [ value *( value-separator value ) ] end-array

array = begin-array [ value *( value-separator value ) ] end-array

There is no requirement that the values in an array be of the same type.

配列内の値が同じ型である必要はありません。

6. Numbers

6. 数値

The representation of numbers is similar to that used in most programming languages. A number is represented in base 10 using decimal digits. It contains an integer component that may be prefixed with an optional minus sign, which may be followed by a fraction part and/or an exponent part. Leading zeros are not allowed.

数値の表現は、一般的なプログラミング言語で使用される表現と同様です。数値は10進数字を使用して10進法で表現されます。整数部分を含み、オプションでマイナス記号を付けることができ、小数部分および/または指数部分が続いてもよいです。先行ゼロは許可されていません。

A fraction part is a decimal point followed by one or more digits.

小数部分は、小数点の後に1つ以上の整数が続きます。

An exponent part begins with the letter E in uppercase or lowercase, which may be followed by a plus or minus sign. The E and optional sign are followed by one or more digits.

指数部分は、大文字または小文字のEで始まり、プラス記号またはマイナス記号が続いてもよいです。Eとオプションの記号の後に1つ以上の整数が続きます。

Numeric values that cannot be represented in the grammar below (such as Infinity and NaN) are not permitted.

下記の文法では表現できない数値（InfinityやNaNなど）は許可されていません。

   number = [ minus ] int [ frac ] [ exp ]


   number = [ minus ] int [ frac ] [ exp ]


   decimal-point = %x2E       ; .


   decimal-point = %x2E       ; .


   digit1-9 = %x31-39         ; 1-9


   digit1-9 = %x31-39         ; 1-9


   e = %x65 / %x45            ; e E


   e = %x65 / %x45            ; e E


   exp = e [ minus / plus ] 1*DIGIT


   exp = e [ minus / plus ] 1*DIGIT


   frac = decimal-point 1*DIGIT
   int = zero / ( digit1-9 *DIGIT )


   frac = decimal-point 1*DIGIT
   int = zero / ( digit1-9 *DIGIT )


   minus = %x2D               ; -


   minus = %x2D               ; -


   plus = %x2B                ; +


   plus = %x2B                ; +


   zero = %x30                ; 0


   zero = %x30                ; 0

This specification allows implementations to set limits on the range and precision of numbers accepted. Since software that implements IEEE 754 binary64 (double precision) numbers [IEEE754] is generally available and widely used, good interoperability can be achieved by implementations that expect no more precision or range than these provide, in the sense that implementations will approximate JSON numbers within the expected precision. A JSON number such as 1E400 or 3.141592653589793238462643383279 may indicate potential interoperability problems, since it suggests that the software that created it expects receiving software to have greater capabilities for numeric magnitude and precision than is widely available.

本仕様では、実装が受け付ける数値の範囲と精度に制限を設定することができます。IEEE 754 64ビット浮動小数点数[IEEE754]を実装したソフトウェアは一般的に利用可能で広く使用されているため、これ以上の精度や範囲を必要としない実装を使用することで、相互運用性を達成できます。つまり、実装はJSON数値を期待される精度の範囲で近似することになります。1E400や3.141592653589793238462643383279といったJSON数値は、潜在的な相互運用性の問題を示す可能性があります。これは、それを作成したソフトウェアが、受信側のソフトウェアに対して一般的に利用可能な範囲より大きな数値の規模と精度の能力を持つことを期待していることを示しているためです。

Note that when such software is used, numbers that are integers and are in the range [-(2**53)+1, (2**53)-1] are interoperable in the sense that implementations will agree exactly on their numeric values.

そういったソフトウェアが使用される場合、整数であって[-(2**53)+1, (2**53)-1]の範囲内にある数値は相互運用性を持ちます。これは、実装がその数値を正確に一致させるという意味です。

7. Strings

7. 文字列

The representation of strings is similar to conventions used in the C family of programming languages. A string begins and ends with quotation marks. All Unicode characters may be placed within the quotation marks, except for the characters that MUST be escaped: quotation mark, reverse solidus, and the control characters (U+0000 through U+001F).

文字列の表現は、C言語ファミリーのプログラミング言語で使用される規則に似ています。文字列は引用符で始まり、引用符で終わります。すべてのUnicode文字を引用符内に配置できますが、エスケープするしなければならない（MUST）文字（引用符、逆斜線、および制御文字U+0000～U+001F）は除外されます。

Any character may be escaped. If the character is in the Basic Multilingual Plane (U+0000 through U+FFFF), then it may be represented as a six-character sequence: a reverse solidus, followed by the lowercase letter u, followed by four hexadecimal digits that encode the character's code point. The hexadecimal letters A through F can be uppercase or lowercase. So, for example, a string containing only a single reverse solidus character may be represented as "\u005C".

任意の文字をエスケープすることができます。文字が基本多言語面（U+0000～U+FFFF）にある場合、6文字のシーケンスとして表現することができます：逆斜線に続いて、小文字のu、その後に文字のコードポイントをエンコードする4桁の16進数字が続きます。16進数字AからFは大文字でも小文字でも構いません。たとえば、逆斜線文字のみを含む文字列は、"\u005C"として表現できます。

Alternatively, there are two-character sequence escape representations of some popular characters. So, for example, a string containing only a single reverse solidus character may be represented more compactly as "\\".

別の方法として、人気のある文字のいくつかには、2文字のシーケンスエスケープ表現があります。たとえば、逆斜線文字のみを含む文字列は、より簡潔に"\\"として表現できます。

To escape an extended character that is not in the Basic Multilingual Plane, the character is represented as a 12-character sequence, encoding the UTF-16 surrogate pair. So, for example, a string containing only the G clef character (U+1D11E) may be represented as "\uD834\uDD1E".

基本多言語面にない拡張文字をエスケープするには、文字はUTF-16サロゲートペアをエンコードする12文字のシーケンスとして表現されます。たとえば、ト音記号文字（U+1D11E）のみを含む文字列は、"\uD834\uDD1E"として表現できます。

8. String and Character Issues

8. 文字列と文字の問題

8.1. Character Encoding

8.1. 文字エンコーディング

JSON text exchanged between systems that are not part of a closed ecosystem MUST be encoded using UTF-8 [RFC3629].

閉じたエコシステムの一部ではないシステム間で交換されるJSON テキストは、しなければなりません（MUST） UTF-8を使用してエンコードされます [RFC3629]。

Previous specifications of JSON have not required the use of UTF-8 when transmitting JSON text. However, the vast majority of JSON- based software implementations have chosen to use the UTF-8 encoding, to the extent that it is the only encoding that achieves interoperability.

JSONの以前の仕様では、JSON テキストの送信時にUTF-8の使用を要求していません。ただし、JSON ベースのソフトウェア実装の大多数はUTF-8エンコーディングを選択しており、相互運用性を達成する唯一のエンコーディングである程度にまで至っています。

Implementations MUST NOT add a byte order mark (U+FEFF) to the beginning of a networked-transmitted JSON text. In the interests of interoperability, implementations that parse JSON texts MAY ignore the presence of a byte order mark rather than treating it as an error.

実装は、してはなりません（MUST NOT） バイト順マーク（U+FEFF）をネットワーク経由で送信されるJSON テキストの開始に追加してください。相互運用性の観点から、JSON テキストを解析する実装は、エラーとして扱うのではなく、バイト順マークの存在をしてもよいです（MAY） 無視してもいいです。

8.2. Unicode Characters

8.2. Unicode 文字

When all the strings represented in a JSON text are composed entirely of Unicode characters [UNICODE] (however escaped), then that JSON text is interoperable in the sense that all software implementations that parse it will agree on the contents of names and of string values in objects and arrays.

JSON テキストで表現されるすべての文字列が Unicode 文字 [UNICODE] のみで構成されている場合（どのようにエスケープされているかに関わらず）、そのJSON テキストは相互運用可能です。つまり、それを解析するすべてのソフトウェア実装は、オブジェクトと配列内の名前と文字列値の内容について同意します。

However, the ABNF in this specification allows member names and string values to contain bit sequences that cannot encode Unicode characters; for example, "\uDEAD" (a single unpaired UTF-16 surrogate). Instances of this have been observed, for example, when a library truncates a UTF-16 string without checking whether the truncation split a surrogate pair. The behavior of software that receives JSON texts containing such values is unpredictable; for example, implementations might return different values for the length of a string value or even suffer fatal runtime exceptions.

ただし、この仕様の ABNF では、メンバー名と文字列値に Unicode 文字をエンコードできないビット列を含めることができます。例えば、"\uDEAD"（ペアになっていない単一の UTF-16 サロゲート）です。このような例は、例えばライブラリが UTF-16 文字列を截断する際に截断がサロゲートペアを分割しているかどうかを確認せずに発生することが観察されています。このような値を含む JSON テキストを受け取るソフトウェアの動作は予測不可能です。例えば、実装は文字列値の長さに異なる値を返すか、致命的な実行時例外さえ発生させる可能性があります。

8.3. String Comparison

8.3. 文字列の比較

Software implementations are typically required to test names of object members for equality. Implementations that transform the textual representation into sequences of Unicode code units and then perform the comparison numerically, code unit by code unit, are interoperable in the sense that implementations will agree in all cases on equality or inequality of two strings. For example, implementations that compare strings with escaped characters unconverted may incorrectly find that "a\\b" and "a\u005Cb" are not equal.

ソフトウェア実装は、通常、オブジェクトのメンバー名が等しいかどうかをテストする必要があります。テキスト表現を Unicode コード単位のシーケンスに変換し、その後コード単位ごとに数値比較を実行する実装は、相互運用可能です。つまり、実装は 2 つの文字列が等しいか等しくないかについてすべての場合に同意します。例えば、エスケープ文字を変換しないで文字列を比較する実装は、"a\\b" と "a\u005Cb" が等しくないと誤って判断する可能性があります。

9. Parsers

9. パーサー

A JSON parser transforms a JSON text into another representation. A JSON parser MUST accept all texts that conform to the JSON grammar. A JSON parser MAY accept non-JSON forms or extensions.

JSON パーサーは JSON テキストを別の表現に変換します。JSON パーサーは JSON 文法に準拠するすべてのテキストを受け入れるしなければなりません（MUST）。JSON パーサーは非 JSON 形式または拡張機能を受け入れるしてもよいです（MAY）。

An implementation may set limits on the size of texts that it accepts. An implementation may set limits on the maximum depth of nesting. An implementation may set limits on the range and precision of numbers. An implementation may set limits on the length and character contents of strings.

実装は、受け入れるテキストのサイズに制限を設定してもかまいません。実装は、ネストの最大深さに制限を設定してもかまいません。実装は、数値の範囲と精度に制限を設定してもかまいません。実装は、文字列の長さと文字内容に制限を設定してもかまいません。

10. Generators

10. ジェネレーター

A JSON generator produces JSON text. The resulting text MUST strictly conform to the JSON grammar.

JSON ジェネレーターは JSON テキストを生成します。生成されたテキストは JSON 文法に厳密に準拠するしなければなりません（MUST）。

12. Security Considerations

12. セキュリティーに関する考慮事項

Generally, there are security issues with scripting languages. JSON is a subset of JavaScript but excludes assignment and invocation.

一般的に、スクリプト言語にはセキュリティー上の問題があります。JSON は JavaScript のサブセットですが、割り当てと呼び出しは除外されています。

Since JSON's syntax is borrowed from JavaScript, it is possible to use that language's "eval()" function to parse most JSON texts (but not all; certain characters such as U+2028 LINE SEPARATOR and U+2029 PARAGRAPH SEPARATOR are legal in JSON but not JavaScript). This generally constitutes an unacceptable security risk, since the text could contain executable code along with data declarations. The same consideration applies to the use of eval()-like functions in any other programming language in which JSON texts conform to that language's syntax.

JSON の構文は JavaScript から借用されているため、その言語の "eval()" 関数を使用してほとんどの JSON テキストを解析することが可能です（ただし、すべてではありません。U+2028 LINE SEPARATOR や U+2029 PARAGRAPH SEPARATOR などの特定の文字は JSON では合法ですが JavaScript では合法ではありません）。これは一般的には受け入れられないセキュリティーリスクを構成しています。テキストにはデータ宣言とともに実行可能なコードが含まれる可能性があるためです。同じ考慮事項は、JSON テキストがその言語の構文に準拠する他のプログラミング言語での eval() のような関数の使用にも適用されます。
